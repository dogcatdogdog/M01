## 1. 引言

### 1.1 目的

本文档是"面向无人机任务执行的MCP控制协议与人机协同异常决策机制"的软件需求规格说明书（SRS），旨在为系统的设计、开发、测试与验收提供完整、明确、可验证的需求基线。

本文档面向以下读者：

- 系统架构师：理解MCP协议架构、人机协同决策机制的技术选型边界
- 协议开发工程师：实现MCP控制协议的编解码、消息路由、状态管理
- AI决策工程师：实现异常检测、决策推理、人机交互决策逻辑
- 后端开发工程师：实现任务执行监控、异常事件处理、工单流转
- 前端开发工程师：实现实时监控界面、人机决策交互面板、告警通知
- 测试工程师：依据本文档制定测试用例和验收标准
- 产品经理与研究人员：作为课题研究与专利申请的需求依据

### 1.2 范围

本系统聚焦以下四个核心能力域：

1. **MCP控制协议**：设计面向无人机任务执行全过程的标准化控制协议，覆盖任务下发、状态上报、指令控制、异常通报四大消息类型，实现地面站、AI决策引擎、无人机飞控之间的统一通信语言。

2. **任务执行状态机管理**：基于MCP协议消息驱动的任务执行全生命周期状态管理，支持状态回溯与断点续飞。

3. **多级异常检测与分类**：构建"飞控层→遥测层→任务层→环境层"四级异常检测体系，融合规则引擎、统计模型和大语言模型，实现全尺度异常覆盖。

4. **人机协同异常决策机制**：提出"AI自主→人机协商→人工接管"三级决策权限模型，在保障安全的前提下最大化AI自主处置能力。

本文档**不覆盖**：无人机底层飞控固件、传感器硬件驱动、视频流实时传输、PHM健康管理系统的规则管理和大模型诊断逻辑。

## 2. 总体描述

### 2.1 产品愿景

无人机任务执行阶段是安全风险最高、异常事件最密集的环节。当前系统存在三个根本性问题：

- **协议碎片化**：不同厂商无人机使用不同控制协议，系统集成成本高、可靠性低
- **异常处置被动化**：异常发生后依赖操作员人工判断，响应时间长（30秒~2分钟）
- **人机权责模糊**：AI自动化与人工干预之间缺乏清晰的分级机制

本系统愿景是构建**"协议标准化、异常智能化、决策协同化"**的新一代任务执行控制范式。目标将异常响应时间从45秒缩短至5秒以内，AI自主处置成功率提升至90%以上，同时确保零安全事故。

**课题创新性**：

1. **MCP控制协议设计**：借鉴Model Context Protocol思想，设计"工具注册→上下文注入→指令调用→结果反馈"标准化控制流程
2. **三级人机协同决策模型**：提出"AI自主（LoA-4）→人机协商（LoA-2/3）→人工接管（LoA-0/1）"动态权限切换模型
3. **OODA循环加速**：将传统OODA循环压缩为"感知→推理→行动"AI快速通道

### 2.2 系统架构概述

系统采用**"协议层→监控层→决策层→交互层"**四层架构：

```
┌─────────────────────────────────────────────────────────────┐
│              人机协同决策交互界面（前端）                       │
│  实时态势 / 异常告警 / 决策建议 / 人工接管 / 信任度调节         │
└─────────────────────────┬───────────────────────────────────┘
                          ▼
┌─────────────────────────────────────────────────────────────┐
│              第四层：人机协同决策引擎                           │
│  三级决策权限模型 / AI自主决策 / 人机协商 / 信任度评估           │
└─────────────────────────┬───────────────────────────────────┘
                          ▼
┌─────────────────────────────────────────────────────────────┐
│              第三层：多级异常检测引擎                           │
│  飞控层(ms) → 遥测层(s) → 任务层(10s) → 环境层(min)           │
│  规则引擎 / 统计模型 / LLM推理 / 异常关联                      │
└─────────────────────────┬───────────────────────────────────┘
                          ▼
┌─────────────────────────────────────────────────────────────┐
│              第二层：任务执行状态机                             │
│  状态管理 / 进度追踪 / 断点续飞 / 多机同步                     │
└─────────────────────────┬───────────────────────────────────┘
                          ▼
┌─────────────────────────────────────────────────────────────┐
│              第一层：MCP控制协议层                              │
│  协议编解码 / 消息路由 / 设备注册 / 会话管理 / 传输适配         │
└─────────────────────────┬───────────────────────────────────┘
                          ▼
┌─────────────────────────────────────────────────────────────┐
│              无人机飞控系统 / 传感器 / 载荷                     │
└─────────────────────────────────────────────────────────────┘
```

**核心设计原则**：
- MCP协议作为系统唯一的任务控制通信语言
- 异常检测分层处理，低层就地处理，高层逐级上报
- 人机协同遵循"最小权限原则"，信任度驱动权限动态升降
- 人工操作员始终保留最高优先级紧急接管权限

## 4. 功能性需求

### 4.1 MCP控制协议设计

#### 4.1.1 需求概述

MCP控制协议借鉴Model Context Protocol的"工具注册→上下文注入→指令调用→结果反馈"范式，设计面向无人机任务执行全过程的标准化消息协议。

#### 4.1.2 协议架构与消息类型

**FR-MCP-001：协议层次结构**

MCP协议采用四层设计：应用层（任务指令/状态报告/异常通报/决策交互）、语义层（消息语义定义/能力声明/上下文描述）、会话层（设备注册/会话管理/心跳保活）、传输层（消息序列化/传输适配/可靠传输）。

**FR-MCP-002：消息类型体系**

| 消息类别 | 方向 | 具体类型 | 说明 |
| -------- | ---- | -------- | ---- |
| **任务控制** | | MISSION.CREATE/START/PAUSE/RESUME/ABORT/UPDATE | GCS→UAV |
| **状态上报** | | STATUS.HEARTBEAT(1Hz)/POSITION(5Hz)/MISSION_PROGRESS/BATTERY(1Hz)/ATTITUDE(10Hz)/SYSTEM | UAV→GCS |
| **异常通报** | | ANOMALY.DETECTED/ESCALATED/RESOLVED/ACTION_REQUEST | UAV→GCS |
| **决策交互** | | DECISION.AI_RECOMMENDATION/HUMAN_OVERRIDE/TRUST_UPDATE/SWITCH_LOA | AI↔GCS↔UAV |

**FR-MCP-003：MCP消息信封结构**

每条MCP消息包含标准化信封：header（协议版本、消息ID、时间戳、源/目标节点、优先级、QoS）、context（任务ID、设备ID、会话ID、自主等级）、payload（消息类型+数据）、trace（跳数、路由路径、延迟预算）。

#### 4.1.3 设备能力注册与会话管理

**FR-MCP-004：设备能力声明**

每架无人机接入MCP网络时声明：支持的命令列表、飞行能力（最大高度/速度/续航/航点数）、传感器能力（相机/热成像/RTK）、通信能力（协议/带宽/链路类型）、自主能力（最大LoA/应急动作/安全保护）。

**FR-MCP-005：能力查询与匹配**

AI决策前查询设备能力，确保动作在能力范围内；超出能力时自动降级为替代动作。

**FR-MCP-006：MCP会话生命周期**

| 状态 | 说明 | 转换条件 |
| ---- | ---- | -------- |
| CONNECTING | 建立连接中 | 设备发起注册 |
| REGISTERED | 已注册，能力声明完成 | 能力验证通过 |
| ACTIVE | 执行任务中 | 任务启动指令 |
| SUSPENDED | 临时离线 | 心跳超时10秒 |
| RECOVERING | 恢复中 | 收到重连消息 |
| TERMINATED | 会话终止 | 主动注销或任务完成 |

**FR-MCP-007：心跳保活机制**

设备侧每1秒发送心跳；3秒未收到标记"可能离线"，10秒标记SUSPENDED。心跳携带电量、GPS状态、飞控模式、任务状态摘要。

#### 4.1.4 传输层设计

**FR-MCP-008/009：消息序列化与可靠性**

默认Protobuf序列化，备选JSON。QoS三级：AT_MOST_ONCE（高频遥测）、AT_LEAST_ONCE（位置/电量）、EXACTLY_ONCE（关键控制指令/人工接管）。

**FR-MCP-010：协议适配层**

提供MAVLink适配器、DJI Cloud API适配器、自定义协议适配器，将底层协议消息双向转换为MCP标准消息。

### 4.2 任务执行状态机

#### 4.2.1 需求概述

任务执行状态机是MCP消息驱动的任务全生命周期管理核心。

#### 4.2.2 状态定义与转换

**FR-MCP-011：任务执行状态机**

系统实现9个状态的状态机：CREATED→PRE_CHECK→TAKING_OFF→EXECUTING→COMPLETED，中间可转PAUSED/ABORTING/LANDING，任意阶段可转FAILED。

| 状态 | MCP触发消息 | 说明 |
| ---- | ----------- | ---- |
| CREATED | MISSION.CREATE | 任务已创建 |
| PRE_CHECK | MISSION.START | 自动检查设备/气象/禁飞区 |
| TAKING_OFF | — | 起飞至航线起始高度 |
| EXECUTING | — | 航线执行中 |
| PAUSED | MISSION.PAUSE | 悬停等待 |
| ABORTING | MISSION.ABORT | 中止并返航 |
| LANDING | — | 降落中 |
| COMPLETED | — | 正常完成 |
| FAILED | — | 异常失败 |

**FR-MCP-012：预检流程**

8项预检：设备在线、电池充足（≥预估消耗+30%）、GPS锁定（≥10卫星）、飞控就绪、气象条件、航线合规、传感器就绪、链路质量（RSSI>-85dBm）。

**FR-MCP-013：断点续飞**

记录中断时精确状态（航点索引、位置、已完成列表）；恢复从未完成航点继续；中断超30分钟需人工确认。

**FR-MCP-014：多机协同状态管理**

维护任务级和设备级两层状态机；通过missionId关联同任务设备状态；完成条件可配置"全部完成"或"任一完成"。

### 4.3 多级异常检测体系

#### 4.3.1 需求概述

构建四级异常检测体系，从毫秒级硬件异常到分钟级环境变化实现全尺度覆盖。

#### 4.3.2 四级异常检测

**FR-MCP-015：飞控层异常（毫秒级）**

| 异常类型 | 检测方式 | 典型响应 |
| -------- | -------- | -------- |
| 电机异常 | 电流/转速偏差 | 飞控自动切换安全模式 |
| IMU异常 | 惯性测量单元自检 | 切换备用传感器 |
| 气压计异常 | 高度数据跳变 | 切换GPS高度源 |

**FR-MCP-016：遥测层异常（秒级）**

| 异常类型 | 检测规则 | 严重程度 |
| -------- | -------- | -------- |
| 电池低电量 | level<30% | WARNING |
| 电池严重低电 | level<15% | CRITICAL |
| 链路信号弱 | rssi<-85dBm | WARNING |
| 链路中断 | 心跳丢失>10s | CRITICAL |
| GPS信号弱 | satellites<8 | WARNING |
| GPS丢失 | fix<2D | CRITICAL |
| 高度偏差 | 偏差>20m | WARNING |

**FR-MCP-017：任务层异常（10秒级）**

| 异常类型 | 检测逻辑 | 严重程度 |
| -------- | -------- | -------- |
| 航点偏离 | 偏离>50m | WARNING |
| 航点超时 | 超时>200% | WARNING |
| 任务停滞 | 无更新>5分钟 | WARNING |
| 禁飞区入侵 | 接近<100m | CRITICAL |
| 航线冲突 | 多机航线交叉 | CRITICAL |

**FR-MCP-018：LLM辅助分析**

复杂异常调用LLM分析：输入异常类型+任务上下文+遥测快照+历史记录；输出根因推断（置信度）、推荐处置动作（含备选）、风险评估、是否需人工介入。

**FR-MCP-019：环境层异常（分钟级）**

| 异常类型 | 检测方式 | 严重程度 |
| -------- | -------- | -------- |
| 气象恶化 | 风速超标 | WARNING |
| 突发降水 | 传感器/API | CRITICAL |
| 临时禁飞区 | NOTAM更新 | CRITICAL |
| 空域冲突 | ADS-B检测 | CRITICAL |

**FR-MCP-020：多源异常关联**

5秒窗口内多异常合并为事件簇；LLM推理因果关系；根因聚类避免重复决策；关联后严重程度可能升级。

### 4.4 人机协同异常决策机制

#### 4.4.1 需求概述

人机协同异常决策机制是本课题核心创新，设计"AI自主→人机协商→人工接管"三级决策权限模型。

#### 4.4.2 自主等级模型

**FR-MCP-021：五级自主等级**

| 等级 | 编码 | 名称 | 决策主体 | 适用场景 |
| ---- | ---- | ---- | -------- | -------- |
| LoA-4 | FULL_AUTO | AI完全自主 | AI独立决策执行 | 低风险（INFO/WARNING，置信度≥90%） |
| LoA-3 | AI_LED | AI主导 | AI决策后通知人类，不等待确认 | 中等风险（WARNING，置信度≥75%） |
| LoA-2 | SHARED | 人机协商 | AI建议+人类确认 | 中高风险（WARNING/CRITICAL） |
| LoA-1 | HUMAN_LED | 人类主导 | 人类决策+AI信息支持 | 高风险（CRITICAL，置信度<50%） |
| LoA-0 | MANUAL | 人工接管 | 人类完全手动 | 紧急情况/AI故障 |

**FR-MCP-022：自主等级动态切换**

三维度加权评估：异常严重程度（40%）、AI决策置信度（35%）、操作员信任度（25%）。

```
LoA_score = severity×0.4 + confidence×0.35 + trust×0.25
最终LoA = round(LoA_score × 4)
```

**FR-MCP-023：操作员信任度管理**

初始50；正确决策+5，成功自主处置+3；错误决策-10，建议被忽略-5；7天无事件衰减至50；范围[10,95]；支持手动调整[20,90]。

#### 4.4.3 AI自主决策引擎

**FR-MCP-024：AI自主决策流程**

LoA-4：异常→推理→评估→执行→日志→事后通知。LoA-3：异常→推理→评估→执行→日志→实时通知（操作员15秒内可覆盖）。

**FR-MCP-025：AI决策推理过程**

情境评估→规则匹配→LLM推理→方案评估→方案选择→指令生成。

**FR-MCP-026：AI自主决策场景库**

| 场景 | 异常 | AI动作 | 最高等级 |
| ---- | ---- | ------ | -------- |
| AUTO-001 | 电量<30% | 自动返航 | LoA-4 |
| AUTO-002 | 电量<15% | 最近点降落 | LoA-4 |
| AUTO-003 | 链路弱 | 降低高度靠近遥控器 | LoA-4 |
| AUTO-004 | 链路中断>10s | 预设返航航线 | LoA-4 |
| AUTO-005 | GPS弱 | 降低高度悬停 | LoA-3 |
| AUTO-006 | GPS丢失 | ATTI模式悬停 | LoA-3 |
| AUTO-007 | 航点偏离>50m | 修正航向返回航线 | LoA-4 |
| AUTO-008 | 禁飞区接近<100m | 立即远离 | LoA-4 |

#### 4.4.4 人机协商决策流程

**FR-MCP-027：协商决策交互**

LoA-2下：异常→AI生成建议→WebSocket推送决策面板→等待操作员响应（30秒超时）。面板展示：异常详情、AI推荐动作（含置信度）、备选方案、决策依据。操作选项：同意AI建议/选择备选/人工接管/超时自动执行。

**FR-MCP-028：决策超时处理**

LoA-2超时30秒→升级LoA-1（执行最安全保守动作）；连续2次超时→建议提升信任度阈值；CRITICAL超时15秒→自动升级LoA-3。

#### 4.4.5 人工接管机制

**FR-MCP-029：紧急接管**

提供物理按钮、软件红色按钮（右上角固定）、快捷键Ctrl+Shift+T；需2秒内二次确认防误触。

**FR-MCP-030：接管后操作模式**

| 模式 | 说明 | 适用场景 |
| ---- | ---- | -------- |
| 摇杆操控 | 虚拟摇杆控制方向高度 | 精细操控 |
| 地图点击 | 点击目标位置飞往 | 大范围调整 |
| 指令输入 | 输入具体指令 | 明确意图 |
| 航线编辑 | 实时修改航线 | 调整路线 |
| 任务终止 | 中止返航/降落 | 无法继续 |

**FR-MCP-031：归还控制权**

点击"归还AI控制"→安全检查→恢复原自主等级→记录决策日志。

#### 4.4.6 决策冲突与安全红线

**FR-MCP-032：决策冲突处理**

安全红线检测（违反则拒绝）→风险告警（增加风险则确认）→决策记录→安全优先（MISSION.ABORT最高优先级）。

**FR-MCP-033：安全红线（不可覆盖）**

| 红线 | 规则 | 动作 |
| ---- | ---- | ---- |
| REDLINE-001 | 电量<5% | 强制最近点降落 |
| REDLINE-002 | 进入禁飞区 | 强制悬停+返航 |
| REDLINE-003 | 链路中断>60s | 预设失联处置 |
| REDLINE-004 | GPS丢失>30s | 紧急降落 |
| REDLINE-005 | 飞行器翻转/失控 | 触发FTS |

### 附录A：MCP消息Protobuf Schema

```protobuf
syntax = "proto3";
package mcp.v1;

message McpEnvelope {
  McpHeader header = 1;
  McpContext context = 2;
  McpPayload payload = 3;
  McpTrace trace = 4;
}

message McpHeader {
  string protocol = 1;
  string version = 2;
  string message_id = 3;
  string correlation_id = 4;
  int64 timestamp_ms = 5;
  NodeInfo source = 6;
  NodeInfo destination = 7;
  Priority priority = 8;
  int32 ttl = 9;
  QoS qos = 10;
}

message NodeInfo {
  string node_id = 1;
  NodeType node_type = 2;
  NodeRole node_role = 3;
}

enum NodeType { GCS = 0; AI_ENGINE = 1; UAV = 2; OPERATOR = 3; }
enum NodeRole { SENDER = 0; RELAY = 1; GATEWAY = 2; }
enum Priority { LOW = 0; NORMAL = 1; HIGH = 2; CRITICAL = 3; }
enum QoS { AT_MOST_ONCE = 0; AT_LEAST_ONCE = 1; EXACTLY_ONCE = 2; }

message McpContext {
  string mission_id = 1;
  string device_id = 2;
  string session_id = 3;
  int32 autonomy_level = 4;
}

message McpPayload {
  string message_type = 1;
  bytes data = 2;
}

message McpTrace {
  int32 hop_count = 1;
  repeated string route_path = 2;
  int32 latency_budget_ms = 3;
}
```
