# 面向无人平台大模型交互的领域安全护栏框架

## 一、课题背景

### 1.1 大模型在无人平台中的角色正在从"辅助"走向"控制"

在无人机/机器人等无人平台中，大语言模型（LLM）不再只是"回答问题"的聊天工具，而是直接参与任务决策与设备控制的核心组件：

| 典型场景 | LLM 输入 | LLM 输出 | 错误后果 |
|----------|---------|----------|----------|
| 语音指令解析 | 操作员语音 → 文本 | 结构化意图 → 路由到功能模块 | 误解指令，执行错误动作 |
| 故障诊断 | 遥测数据 + 设备档案 | 根因推断 + 维护建议 + 自动创建工单 | 误判故障，延误维修 |
| 任务规划 | 自然语言任务描述 | 航线方案 + 资源调配 + 应急预案 | 生成不安全航线 |
| 异常决策 | 异常事件 + 飞行状态 | 处置动作（返航/悬停/降落） | 错误决策导致坠机 |
| 日志总结 | 飞行记录 + 传感器数据 | 任务报告 + 设备评估 | 信息遗漏，误导后续决策 |

**当 LLM 的输出可以直接触发物理动作时，一次注入攻击或一次幻觉就不仅仅是"聊天记录里多了一句错话"，而可能是一架飞机的坠毁。**

### 1.2 现有 LLM 护栏方案在无人平台场景下存在空白

业界已有成熟的 LLM 安全护栏框架，其 2025-2026 年的生产实践已形成共识架构：**Input Rail → Retrieval Rail → Output Rail → Tool-Call Rail** 四道防线，辅以确定性规则（正则）→ 轻量模型（100M 级）→ 重型模型（8B 级）三级检测流水线，以及并行执行策略（延迟 = 最慢 guard，非所有 guard 之和）。

三个标杆框架的架构分析：

**NVIDIA NeMo Guardrails（⭐4.5k+）**
- 五道护栏：Input / Dialog / Retrieval / Execution / Output Rails
- 自研 Colang DSL 定义安全规则，Python 风格语法
- 输出侧支持 `self check facts` 和 `self check hallucination`（LLM 自检）
- jailbreak 检测内置，PII 脱敏可选
- **局限**：针对对话系统设计，Colang 对无人平台控制流过重

**ProtectAI llm-guard（⭐2k+）**
- Pipeline 模式：15 个 Input Scanner + 21 个 Output Scanner 可组合
- 核心注入检测基于微调的 DeBERTa 模型
- 支持 "redact-and-restore"：输入侧脱敏 → LLM → 输出侧还原
- `BanCode`、`Secrets`、`Gibberish` 等场景化扫描器
- **局限**：36 个 Scanner 全部是通用内容安全，无物理/领域维度

**Guardrails AI（⭐4k+）**
- Pydantic Schema 校验 LLM 输出（与 Function Calling 等价）
- `on_fail` 策略：EXCEPTION / FIX / REASK / FILTER 四种失败处理
- Guardrails Hub — 社区贡献验证器市场
- **局限**：验证器全部是通用内容安全（毒性/竞品/正则），无领域验证器

三个框架的能力覆盖：

| 能力 | NeMo | llm-guard | Guardrails AI |
|------|:--:|:--:|:--:|
| Prompt Injection 检测 | ✅ | ✅ | ❌ |
| PII 脱敏 | ✅ | ✅ | ❌ |
| JSON Schema 强制 | ❌ | ✅ | ✅ |
| 幻觉/事实检测 | ✅ | ✅ | ❌ |
| 工具调用安全检查 | ✅ | ❌ | ❌ |
| **领域物理量值域校验** | ❌ | ❌ | ❌ |
| **安全边界规则引擎** | ❌ | ❌ | ❌ |
| **上下文涉密分级脱敏** | ❌ | ❌ | ❌ |
| **不可篡改审计日志** | ❌ | ❌ | ❌ |
| **safe-fail（不确定即拒绝）** | 部分 | ❌ | ✅ |

这些方案的设计出发点是**聊天、客服、内容生成**场景。它们不知道"飞行高度超出限高"是危险的，不认识 GPS 坐标需要脱敏，不理解"禁飞区"是什么。在无人平台场景下存在具体空白：

| 维度 | 通用护栏方案 | 无人平台的需求 |
|------|:---:|------|
| 输入威胁模型 | 越狱提示词、仇恨言论 | + 军事坐标伪装注入、语音信道噪声攻击 |
| 输出校验深度 | JSON 格式合法性 | + 物理量值域校验（高度/速度/电量是否在物理可能范围内）、航点是否在禁飞区外 |
| 上下文处理 | 全量进入 LLM | 涉密数据（精确坐标、人员身份、设备编号）必须预过滤，只送脱敏摘要 |
| 失败模式 | 返回错误文本或 403 | 必须 **safe-fail**：不确定时拒绝执行，向人类操作员预警 |
| 审计要求 | 通常可选 | **强制不可篡改**：每次 LLM 调用的输入/输出/校验结果/决策动作完整留存 |
| 实时性约束 | 无 | 异常决策链路端到端 < 10 秒（含 LLM 推理 + 安全校验） |

### 1.3 已有工程实践验证了需求的真实性和复用价值

在课题组已有的 6 个无人平台 LLM 应用项目中，**"输入校验 → 输出控制 → 安全兜底"**的安全逻辑在每一个项目中以不同形态被重复实现：

| 已有项目 | 场景 | 已实现的安全措施（碎片化） |
|----------|------|--------------------------|
| **M01 意图解析** | 语音 → 意图分类 | 结构异常检测 + Function Calling Schema 强制 + 四层防御提示词 |
| **故障诊断服务** | 遥测 → 诊断报告 | 遥测数据脱敏 + FC JSON Schema + 诊断结果置信度校验 |
| **战术意图解析** | 指令 → 作战方案 | 非法指令拦截 + 目标/单位合法性校验 + 结构化输出约束 |
| **飞行日志总结** | 日志 → 任务报告 | 日志字段过滤 + 敏感信息剥离 |
| **语音识别** | 语音 → 文本 | 噪声/注入音频处理 |

这些项目暴露了同一个结构性问题：**安全代码与业务代码耦合，无法复用，每个新项目都需要重写**。

## 二、核心痛点

1. **碎片化重复**：6 个项目各自实现了相似但互不兼容的输入校验和输出控制逻辑，维护成本随项目数量线性增长。

2. **覆盖不全**：现有实现集中在 Prompt Injection 防御（输入侧），对 **输出侧的安全边界校验**（如 LLM 生成的航点坐标是否在禁飞区内、输出的高度值是否超出设备物理上限）缺乏系统化支持。

3. **涉密信息泄露风险**：无人平台涉及精确 GPS 坐标、设备编号、基站位置、人员身份等涉密数据，当前这些数据以不同程度进入 LLM 的上下文窗口，缺乏统一的数据分级与脱敏策略。

4. **无可审计性**：LLM 的调用决策链路（输入了什么、为什么输出这个、经过了哪些安全检查）无法事后追溯，在安全事件发生时无法定责。

5. **缺乏场景适配的评估基准**：通用护栏方案用 toxicity/jailbreak 指标评估，无法回答"这个护栏能否防止 LLM 生成一条穿越禁飞区的航线"这类领域特定问题。

## 三、研究内容

### 总体目标

构建一套**面向无人平台大模型交互的领域安全护栏框架**，为所有无人平台 LLM 调用提供统一的**输入清洗 → 上下文隔离 → 输出校验 → 审计追溯**四阶段安全保障。框架与具体 LLM 提供商及业务场景解耦，可独立部署或作为库嵌入。

### 方向一：无人平台 LLM 交互威胁建模

- 系统分析无人平台中 LLM 的典型部署位置与交互路径
- 构建区别于通用 LLM 安全的**领域威胁模型**：注入攻击（含语音信道）、涉密数据泄露、物理安全边界突破、幻觉驱动的危险指令
- 建立**无人平台 LLM 安全评估基准**：设计覆盖物理安全维度的对抗测试用例集（不限于文本注入，涵盖坐标级精度、物理量值域、安全红线规则）

### 方向二：输入清洗与上下文隔离机制

- 设计**多级输入检测流水线**：确定性规则（结构异常、Unicode 攻击、协议标记）→ 轻量语义模型 → 领域专用检测器，按延迟预算逐级升级
- 设计**上下文数据分级模型**：定义涉密字段标记体系，LLM 调用前自动剥离不可外传的精确实体值（GPS 坐标 → 区域描述、人员姓名 → 角色代号），仅向 LLM 发送"最小必要信息摘要"
- 支持**不同安全等级任务的差异化配置**：日常巡检 vs 应急响应 vs 涉密作战

### 方向三：输出安全校验与物理边界约束

- 设计**多层次输出校验器**：
  - 格式层：Schema 强制（Function Calling / JSON Schema）
  - 语义层：输出内容与原始意图的一致性校验
  - 物理层：输出的数值是否在物理合理范围内（高度、速度、电量、航程）
  - 安全层：输出的动作是否违反安全红线（禁飞区、限高、设备能力边界）
- 设计 **safe-fail 决策模型**：校验不通过时，不确定即拒绝，向操作员发出结构化预警而非静默降级

### 方向四：可信审计与决策追溯

- 设计**不可篡改的 LLM 调用审计日志结构**：每次调用的完整输入/输出/安全校验结果/最终决策动作
- 实现**决策回溯能力**：安全事件发生后，可完整还原"LLM 看到了什么、输出了什么、经过了哪些安全关卡、谁做了最终决策"
- 支持与现有安全管理体系（工单、飞控日志）的对接

### 方向五：框架工程化与现有项目迁移验证

- 将框架应用于课题组已有 6 个 LLM 项目，替代各项目中的碎片化安全代码
- 验证**代码复用率**（安全相关代码从 N 套收敛为 1 套框架 + 配置）和**安全覆盖完整性**（是否所有项目通过原有对抗测试 + 新增物理安全测试）
- 输出框架 SDK + 配置模板 + 适配指南

## 五、与已有工作的关系

本课题**不是**从零开始。M01 项目（意图解析模块）已经验证了核心架构的核心链路——四层防御（代码层 + API Schema 层 + 提示词层 + 兜底层）在真实 LLM（Qwen-turbo）上通过 144 个测试用例，其中 58 个对抗测试覆盖注入攻击、安全边界、歧义识别、多指令冲突。本课题将这一经验从单一意图解析场景抽象为通用框架，并补充上下文隔离、物理边界校验和审计追溯能力。

## 六、摘要

**课题背景：** 大模型在无人平台中正从"聊天辅助"走向"任务控制"——LLM 的输出可以直接触发物理动作（返航、降落、航线变更），但业界现有的 LLM 安全护栏（NeMo Guardrails、llm-guard、Guardrails AI）全部针对聊天/客服场景设计，缺乏物理安全边界校验、涉密数据分级脱敏和不可篡改审计能力，且课题组 6 个 LLM 项目中相似的安全逻辑以碎片化方式重复实现。

**课题方向：** 构建一个面向无人平台的、独立于业务与 LLM 提供商的领域安全护栏框架，统一提供输入清洗（注入检测+坐标脱敏）→ 上下文隔离（涉密字段最小化摘要）→ 输出校验（Schema + 物理量值域 + 安全红线）→ 审计追溯（不可篡改全链路日志）四阶段安全保障，并以已有 6 个项目为迁移对象验证框架的复用性与安全覆盖完整性。
