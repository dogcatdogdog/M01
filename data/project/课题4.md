## 1. 引言

### 1.1 目的

本文档是"基于自然语言意图识别的五维推理链无人机任务规划方法"的软件需求规格说明书（SRS），旨在为系统的设计、开发、测试与验收提供完整、明确、可验证的需求基线。

本文档面向以下读者：

- 系统架构师：理解五维推理链的架构约束与模型选型边界
- NLP/AI工程师：实现意图识别引擎、推理链各维度模型、提示词工程
- 后端开发工程师：实现任务规划编排引擎、推理流水线、知识库管理
- 前端开发工程师：实现自然语言交互界面、推理过程可视化、任务预览
- 测试工程师：依据本文档制定测试用例和验收标准
- 产品经理与研究人员：作为课题研究与专利申请的需求依据

### 1.2 范围

本系统聚焦以下五个核心能力域：

1. **自然语言意图识别**：接收用户以自然语言描述的无人机任务需求，通过大语言模型进行意图解析，提取任务目标、约束条件、优先级偏好等结构化语义信息，支持多轮对话式意图澄清与细化。

2. **五维推理链任务规划**：构建"意图解析维 → 环境感知维 → 约束推理维 → 路径规划维 → 资源调配维"的五级推理链，每一维度以前一维度的输出为输入，逐步将抽象自然语言意图转化为可执行的任务计划，实现从语义到执行的全链路推理。

3. **环境态势感知与推理**：实时融合气象数据、禁飞区信息、地形数据、设备状态、历史巡检记录等多源异构信息，为推理链提供实时环境上下文，支持环境变化下的任务动态调整。

4. **约束满足与优化推理**：将安全约束（禁飞区、限高、最小安全距离）、设备约束（续航、载荷、传感器能力）、任务约束（时间窗口、优先级、覆盖要求）统一建模为约束满足问题，通过推理引擎自动求解满足所有约束的可行方案。

5. **任务规划输出与验证**：将推理结果转化为包含航线、设备分配、时间计划、应急预案的完整任务方案，提供可视化预览与可行性验证，支持人工确认或修改后一键下发执行。

本文档**不覆盖**以下内容：实时飞行控制、遥测数据采集、图像AI检测、设备硬件管理、PHM健康管理。

## 2. 总体描述

### 2.1 产品愿景

当前无人机任务规划存在三个根本性瓶颈：

- **高门槛**：任务规划依赖操作人员掌握航线编辑器、GIS工具和无人机专业知识，学习成本高，非专业人员无法参与任务规划决策；
- **静态化**：任务规划通常为"一次规划、全程执行"模式，无法根据实时环境变化（天气突变、新禁飞区生效、设备异常）动态调整任务方案；
- **碎片化**：意图理解、环境感知、约束检查、路径生成、资源分配五个环节各自独立，缺乏统一推理框架串联，导致规划结果割裂、优化不充分。

本系统的愿景是构建**"一句话规划、全链路推理、自适应执行"**的新一代智能任务规划范式。用户只需用自然语言描述任务意图（如"明天上午用三号机对工业园区东侧进行巡检，重点检查烟囱和管道，飞行高度不超过120米"），系统通过五维推理链自动生成优化的可执行任务方案，将任务规划时间从平均45分钟缩短至3分钟以内，将规划成功率（无需人工修改即可直接执行）提升至85%以上。

**课题创新性**：

1. **五维推理链架构**：创新性地将任务规划分解为五个语义维度的串联推理，每个维度对应一个独立的推理模块，维度间通过结构化中间表示传递信息，实现"每一步可解释、每一层可审计"的透明推理过程；
2. **自然语言驱动**：首次将大语言模型的语义理解能力深度嵌入任务规划流程，使非专业用户也能通过对话方式完成复杂任务规划；
3. **环境自适应推理**：环境感知维实时融合多源异构数据，推理链在环境变化时可自动回溯到受影响维度重新推理，而非从头开始。

### 2.2 系统架构概述

五维推理链任务规划采用**"串行推理 + 条件回溯"**流水线架构：

```
┌──────────────────────────────────────────────────────────────────────────┐
│                     自然语言交互界面（前端）                                │
│  用户输入任务描述 / 多轮对话澄清 / 推理过程可视化 / 方案预览与确认           │
└───────────────────────────┬──────────────────────────────────────────────┘
                            │ 原始自然语言输入
                            ▼
┌──────────────────────────────────────────────────────────────────────────┐
│            第一维：意图解析引擎（Intent Parsing Engine）                    │
│  NLU解析 → 意图分类 → 实体抽取 → 约束提取 → 结构化意图JSON输出              │
│  ↕ 多轮对话：置信度 < 阈值时生成澄清问题，与用户交互完善意图                │
└───────────────────────────┬──────────────────────────────────────────────┘
                            │ 结构化意图对象
                            ▼
┌──────────────────────────────────────────────────────────────────────────┐
│            第二维：环境感知引擎（Environment Perception Engine）            │
│  气象数据融合 → 禁飞区/限高区叠加 → 地形数据加载 → 设备状态感知             │
│  → 环境风险评估 → 环境约束图生成                                           │
└───────────────────────────┬──────────────────────────────────────────────┘
                            │ 环境约束图 + 结构化意图
                            ▼
┌──────────────────────────────────────────────────────────────────────────┐
│            第三维：约束推理引擎（Constraint Reasoning Engine）              │
│  安全约束校验 → 设备约束匹配 → 时空约束推理 → 约束冲突检测与消解             │
│  → 可行解空间定义 → 约束满足度评估                                         │
└───────────────────────────┬──────────────────────────────────────────────┘
                            │ 约束满足方案集 + 可行解空间
                            ▼
┌──────────────────────────────────────────────────────────────────────────┐
│            第四维：路径规划引擎（Path Planning Engine）                     │
│  区域覆盖算法 → 航线生成 → 航点优化 → 禁飞区规避 → 高度剖面规划             │
│  → 时间/能耗估算 → 路径代价评估                                            │
└───────────────────────────┬──────────────────────────────────────────────┘
                            │ 候选航线方案集
                            ▼
┌──────────────────────────────────────────────────────────────────────────┐
│            第五维：资源调配引擎（Resource Allocation Engine）               │
│  设备匹配评估 → 多机任务分解 → 时间窗口调度 → 备份方案生成                  │
│  → 应急预案编排 → 最终任务计划输出                                          │
└───────────────────────────┬──────────────────────────────────────────────┘
                            │ 完整任务计划
                            ▼
┌──────────────────────────────────────────────────────────────────────────┐
│                  任务计划管理（Task Plan Management）                      │
│  可视化预览 / 人工审核 / 修改反馈 / 一键下发至任务管理模块                   │
└──────────────────────────────────────────────────────────────────────────┘
```

**核心设计原则**：

- 每个推理维度独立封装，拥有独立的提示词模板、知识库和推理策略，可单独迭代升级；
- 维度间通过结构化JSON中间表示通信，避免自然语言传递的语义损耗；
- 任一维度推理失败或置信度不足时，可向前一维度回溯请求补充信息；
- 推理全程生成审计日志，每一维度的输入、输出、推理依据可追溯。

## 4. 功能性需求

### 4.1 自然语言意图识别功能

#### 4.1.1 需求概述

系统应提供自然语言任务描述的意图解析能力，支持用户以口语化、非结构化的文本输入描述巡检任务需求，通过大语言模型将自然语言转化为结构化的任务意图对象。

#### 4.1.2 意图输入模型

**FR-NLP-001：自然语言任务描述输入**

系统应支持以下输入方式：

- **文本输入**：用户在任务规划页面的文本输入框中输入自然语言描述（支持中英文混合），长度限制为10~2000字符
- **语音输入**：支持语音转文本后输入（预留接口，首期可仅实现文本）
- **多轮对话**：系统对模糊或不完整的输入生成澄清问题，用户回答后系统合并上下文重新解析

**FR-NLP-002：意图解析输出结构**

系统应从自然语言输入中提取并输出以下结构化意图对象：

| 字段名 | 类型 | 说明 | 示例 |
| ------ | ---- | ---- | ---- |
| intentId | UUID | 意图对象唯一标识 | — |
| missionType | Enum | 任务类型：INSPECTION/SURVEY/MONITORING/DELIVERY/EMERGENCY | INSPECTION |
| targetArea | Object | 目标区域描述（支持自然语言地名、坐标范围、行政区划） | "工业园区东侧" |
| targetPoints | List\<Object\> | 具体巡检目标点（建筑物、设施等） | ["烟囱A", "管道B"] |
| priority | Enum | 任务优先级：URGENT/HIGH/MEDIUM/LOW | MEDIUM |
| timeWindow | Object | 时间窗口：期望执行时间/时间范围/周期频率 | "明天上午9-11点" |
| devicePreference | Object | 设备偏好：指定设备/设备型号/数量要求 | "三号机" 或 "至少2台Mavic3E" |
| altitudeConstraint | Object | 高度约束：最大/最小高度/高度范围 | "不超过120米" |
| coverageRequirement | Object | 覆盖要求：全覆盖/重点区域/指定目标 | "重点检查烟囱和管道" |
| qualityRequirement | Object | 质量要求：图像分辨率/重叠率/拍摄角度 | "高清拍摄，重叠率80%" |
| weatherSensitivity | Enum | 天气敏感度：HIGH/MEDIUM/LOW | HIGH |
| specialConstraints | List\<String\> | 其他特殊约束（自然语言描述） | "避开居民区上方" |
| confidence | Float | 意图解析置信度（0~1） | 0.87 |
| ambiguityFlags | List\<Object\> | 模糊项标记（需澄清的字段） | [{"field":"timeWindow","question":"具体是几点到几点？"}] |
| rawInput | String | 原始用户输入 | "明天上午用三号机对工业园区东侧…" |
| sessionId | String | 对话会话ID（多轮对话关联） | — |

#### 4.1.3 意图分类与实体抽取

**FR-NLP-003：任务意图分类**

系统应支持以下任务意图类型分类：

| 意图类型 | 编码 | 典型输入示例 | 关键推理特征 |
| -------- | ---- | ------------ | ------------ |
| 巡检任务 | INSPECTION | "对XX区域进行日常巡检" | 区域覆盖、航点遍历、图像采集 |
| 测量任务 | SURVEY | "对XX工地进行三维建模" | 精确坐标、倾斜摄影、高重叠率 |
| 监控任务 | MONITORING | "对XX区域进行24小时监控" | 定点悬停、周期巡逻、告警触发 |
| 投送任务 | DELIVERY | "将物资运送到XX位置" | 点对点、载荷约束、精确降落 |
| 应急任务 | EMERGENCY | "XX区域发生火灾，立刻起飞监控" | 最短响应时间、优先级最高 |

**FR-NLP-004：命名实体识别（NER）**

系统应从自然语言输入中识别并提取以下类型的实体：

| 实体类型 | 说明 | 示例 |
| -------- | ---- | ---- |
| LOCATION | 地理位置（地名、建筑名、坐标） | "工业园区"、"东经116.4°" |
| DEVICE | 无人机设备标识 | "三号机"、"Mavic 3E" |
| TIME | 时间表达 | "明天上午"、"下周一"、"14:30" |
| ALTITUDE | 高度表达 | "120米"、"100米以上" |
| DURATION | 持续时间 | "两小时"、"一整天" |
| QUANTITY | 数量表达 | "三台"、"两个架次" |
| OBJECT | 巡检目标物 | "烟囱"、"管道"、"光伏板" |
| CONSTRAINT | 约束条件 | "不要飞过居民区"、"保持安全距离" |
| PRIORITY | 优先级表达 | "紧急"、"尽快"、"不着急" |

#### 4.1.4 多轮对话意图澄清

**FR-NLP-005：模糊意图澄清机制**

当意图解析置信度低于可执行阈值时，系统应自动生成澄清问题：

- **置信度阈值**：整体confidence < 0.75 或关键字段（targetArea、missionType）置信度 < 0.85 时触发澄清
- **澄清策略**：
  - 一次最多提出3个澄清问题，避免用户疲劳
  - 问题按字段重要性排序：任务目标 > 区域/目标点 > 时间窗口 > 设备偏好 > 约束条件
  - 问题应具体明确，提供选项或示例（如"您希望在什么时间段执行？例如：明天上午9:00-11:00"）
- **上下文合并**：用户回答澄清后，系统将新信息与原始输入合并，重新执行意图解析
- **最大轮次**：单次任务规划的澄清对话不超过5轮

**FR-NLP-006：意图解析历史与学习**

- 系统应记录每次意图解析的输入、输出和用户确认/修改结果
- 对用户多次修改同一类型任务规划的模式进行统计分析
- 高频出现的意图-修正对可标记为"学习样本"，用于后续模型微调（人工确认后生效）

### 4.2 第一维：意图解析维

#### 4.2.1 需求概述

意图解析维是五维推理链的入口，负责将非结构化的自然语言输入转化为结构化的意图表示。该维度的核心创新在于采用"分类 → 抽取 → 校验 → 补全"四步解析流水线，每步独立可解释。

#### 4.2.2 解析流水线

**FR-NLP-009：意图分类阶段**

- 输入：用户原始自然语言文本
- 处理：调用LLM进行意图分类，输出missionType及置信度
- 输出：分类结果 + 置信度
- 决策规则：
  - 置信度 ≥ 0.85 → 直接进入实体抽取阶段
  - 置信度 0.60~0.85 → 生成1个确认问题（如"您是要进行区域巡检还是定点监控？"）
  - 置信度 < 0.60 → 生成2~3个澄清问题，暂停推理链

**FR-NLP-010：实体抽取阶段**

- 输入：原始文本 + 意图分类结果（作为上下文约束）
- 处理：调用LLM执行命名实体识别，结合GIS地名数据库进行实体标准化
- 输出：实体列表（含类型、原始文本、标准化值、置信度）
- 地名标准化：对于自然语言地名（如"工业园区东侧"），通过GIS知识库匹配为地理坐标范围

**FR-NLP-011：约束提取阶段**

- 输入：原始文本 + 已识别实体列表
- 处理：识别显式约束（"高度不超过120米"）和隐式约束（"对居民区敏感"→ 远离居民区飞行）
- 输出：约束列表（硬约束/软约束、约束类型、约束值）
- 约束分类：
  - 硬约束：不可违反（如禁飞区、最大续航、法规限高）
  - 软约束：尽量满足（如偏好时间段、偏好设备）

**FR-NLP-012：意图校验与补全阶段**

- 输入：完整意图JSON
- 处理：检查必填字段完整性，对缺失的必要字段根据上下文推断补全
- 补全策略：
  - missionType缺失 → 根据关键词推断（"巡检"→INSPECTION）
  - targetArea缺失 → 若输入了具体目标物则推断其所在区域
  - timeWindow缺失 → 默认为"尽快执行"（urgency=HIGH）
  - altitudeConstraint缺失 → 使用法规默认限高（120米）
  - weatherSensitivity缺失 → 根据任务类型默认（SURVEY→HIGH，INSPECTION→MEDIUM）

### 4.3 第二维：环境感知维

#### 4.3.1 需求概述

环境感知维接收意图解析维输出的结构化意图，实时融合多源环境数据，生成环境约束图，为后续约束推理提供真实的物理环境上下文。

#### 4.3.2 环境数据融合

**FR-NLP-013：气象数据融合**

系统应集成气象数据源，为目标区域提供以下气象信息：

| 气象参数 | 说明 | 影响推理 |
| -------- | ---- | -------- |
| 风速/风向 | 实时风速与预报 | 影响航线方向优化、电池消耗估算 |
| 降水概率 | 未来6小时降水预报 | 低于天气敏感度阈值时标记不可飞 |
| 能见度 | 当前能见度 | 低于3km时标记视觉飞行限制 |
| 温度 | 环境温度 | 影响电池性能、设备工作范围 |
| 湿度 | 相对湿度 | 影响设备防护等级要求 |

气象数据更新频率：实时数据每15分钟刷新，预报数据每小时刷新。

**FR-NLP-014：空域约束数据融合**

系统应加载并叠加以下空域约束信息：

- **禁飞区（No-Fly Zone）**：系统标注管理模块维护的永久/临时禁飞区
- **限高区（Altitude Limit Zone）**：不同区域的最大飞行高度限制
- **机场净空区**：机场周边的限高梯度区域
- **临时空域限制**：NOTAM（航行通告）中的临时限制区
- **人口密集区**：基于GIS数据识别的居民区、学校、医院等敏感区域

**FR-NLP-015：地形数据融合**

系统应加载目标区域的地形数据，提供以下信息：

- 地面高程数据（DEM，精度≤30米）
- 地表覆盖类型（建筑/水域/植被/道路/空地）
- 障碍物标注（高大建筑物、电力塔、烟囱等）

**FR-NLP-016：设备状态感知**

系统应实时获取目标设备的当前状态，包括：

- 在线/离线状态
- 当前电量与预估可用飞行时间
- 当前位置与到目标区域的距离
- 设备健康状态（DHI评分，来自PHM系统）
- 最近一次飞行任务间隔时间

#### 4.3.3 环境风险评估

**FR-NLP-017：环境风险评估输出**

环境感知维应将所有环境信息综合为环境约束图（Environment Constraint Map），输出包含：

| 输出字段 | 类型 | 说明 |
| -------- | ---- | ---- |
| weatherCondition | Enum | 飞行适宜性：SUITABLE/MARGINAL/UNSUITABLE |
| weatherDetail | Object | 详细气象数据（各参数当前值与预报值） |
| airspaceConstraints | List\<Object\> | 空域约束列表（类型、几何形状、限制参数） |
| terrainProfile | Object | 目标区域地形剖面数据 |
| obstacleList | List\<Object\> | 障碍物列表（位置、高度、类型） |
| deviceAvailability | List\<Object\> | 可用设备列表及状态详情 |
| environmentRiskScore | Float | 综合环境风险评分（0~1，0为最安全） |
| riskFactors | List\<String\> | 风险因素清单（供后续推理维使用） |
| recommendedFlightWindow | Object | 推荐飞行时间窗口 |
| noGoReasons | List\<String\> | 如评估为UNSUITABLE，列出不可飞行的原因 |

### 4.4 第三维：约束推理维

#### 4.4.1 需求概述

约束推理维是五维推理链的核心智能层，接收意图解析维的结构化意图和环境感知维的环境约束图，将所有约束统一建模，通过约束满足推理求解可行的任务方案空间。

#### 4.4.2 约束分类与建模

**FR-NLP-018：安全约束（硬约束）**

| 约束类型 | 约束说明 | 约束表达 |
| -------- | -------- | -------- |
| 禁飞区规避 | 航线不得穿越任何启用的禁飞区 | 航线几何 ∩ 禁飞区几何 = ∅ |
| 限高约束 | 飞行高度不超过限高区限制 | ∀航点, height ≤ maxAltitude(zone) |
| 安全离地高度 | 飞行高度不低于最低安全高度 | ∀航点, height ≥ minSafetyAltitude |
| 安全距离 | 航线距障碍物最小水平距离 | dist(waypoint, obstacle) ≥ minSafeDist |
| 人员安全 | 航线下方人员密度不超过阈值 | population_density(route_buffer) ≤ threshold |
| 空域授权 | 特殊空域需预先获得飞行许可 | 若目标区域在受限空域，检查授权状态 |

**FR-NLP-019：设备约束（硬约束）**

| 约束类型 | 约束说明 | 约束表达 |
| -------- | -------- | -------- |
| 续航约束 | 任务总时长不超过设备可用飞行时间 | taskDuration ≤ batteryEndurance × safetyFactor |
| 载荷约束 | 任务所需传感器/设备需被搭载 | requiredSensors ⊆ devicePayload |
| 飞行速度 | 设备实际飞行速度在安全范围内 | cruiseSpeed ∈ [minSpeed, maxSpeed] |
| 转弯性能 | 航线转弯半径不低于设备最小转弯半径 | turnRadius ≥ deviceMinTurnRadius |
| 飞控协议 | 设备支持的航点数量不超出限制 | waypointCount ≤ deviceMaxWaypoints |
| 环境适应性 | 设备工作温度范围 | ambientTemp ∈ [minOpTemp, maxOpTemp] |

**FR-NLP-020：任务约束（硬约束/软约束）**

| 约束类型 | 约束说明 | 硬/软 |
| -------- | -------- | ----- |
| 时间窗口 | 任务必须在指定时间窗口内执行 | 硬 |
| 覆盖率 | 巡检区域覆盖率不低于指定百分比 | 硬 |
| 重叠率 | 相邻航线照片重叠率不低于指定值 | 软 |
| 分辨率 | GSD（地面采样距离）满足检测精度要求 | 硬 |
| 多机协同时间约束 | 多机任务时各架次的时序关系 | 硬 |
| 优先级排序 | 多目标时优先级高的先巡检 | 软 |

**FR-NLP-021：约束冲突检测与消解**

当约束之间存在冲突时，系统应执行以下消解策略：

- **冲突检测**：遍历所有约束对，检测是否存在矛盾（如用户要求150米高度但限高为120米）
- **消解优先级**：安全约束 > 设备约束 > 任务约束（硬约束）> 用户偏好（软约束）
- **冲突报告**：生成冲突说明，告知用户哪些约束存在矛盾，建议放宽哪些软约束
- **自动放松**：对软约束按优先级依次放松，直到找到可行解空间

#### 4.4.3 约束求解输出

**FR-NLP-022：约束满足方案输出**

约束推理维的输出应包含：

| 输出字段 | 类型 | 说明 |
| -------- | ---- | ---- |
| feasibleSpace | Object | 可行解空间定义（飞行高度范围、时间窗口范围、可选区域） |
| hardConstraintSet | List\<Object\> | 所有硬约束及其满足状态 |
| softConstraintSet | List\<Object\> | 所有软约束及其满足/放松状态 |
| conflicts | List\<Object\> | 检测到的约束冲突（如有） |
| conflictResolutions | List\<Object\> | 冲突消解方案（如有） |
| feasibleDeviceList | List\<String\> | 满足设备约束的可用设备列表 |
| feasibleTimeWindows | List\<Object\> | 满足时间约束的可执行时间窗口 |
| feasibleAreaPolygon | GeoJSON | 满足所有空间约束后的可飞行区域 |
| constraintSatisfactionScore | Float | 约束满足度评分（0~1） |

### 4.5 第四维：路径规划维

#### 4.5.1 需求概述

路径规划维接收约束推理维定义的可行解空间，根据任务类型和目标区域，生成优化的飞行航线。该维度是五维推理链中计算最密集的环节，需要在约束空间内搜索近似最优路径。

#### 4.5.2 航线生成算法

**FR-NLP-023：区域覆盖航线生成**

针对巡检/测量类任务，系统应支持以下覆盖航线生成策略：

| 策略名称 | 适用场景 | 算法说明 |
| -------- | -------- | -------- |
| 平行扫描（Lawnmower） | 矩形/凸多边形区域全覆盖 | 等间距平行航线，支持自定义航线角度 |
| 向心螺旋 | 圆形/椭圆目标区域 | 从外向内螺旋覆盖 |
| 廊道巡检 | 管道/电力线/道路 | 沿目标物两侧交错航线 |
| 重点目标驻留 | 指定具体目标物 | 在目标物上方多角度环绕拍摄 |
| 混合策略 | 复杂场景 | 根据子区域形状自动选择最优策略 |

**FR-NLP-024：航线参数优化**

系统应对生成的航线进行以下优化：

- **航线方向优化**：根据风向计算最优航线方向，减少侧风飞行段
- **航点间距优化**：根据目标GSD和相机参数计算最优航点间距
- **转弯优化**：优化转弯段航点，减少转弯时间并保持相机指向
- **高度剖面优化**：根据地形高程变化自适应调整飞行高度，保持相对地表恒定高度
- **能量优化**：在满足覆盖要求的前提下，优化航线减少总飞行距离和时间

**FR-NLP-025：禁飞区规避路径生成**

当航线与禁飞区或障碍物存在交集时，系统应自动生成规避路径：

- 检测航线与禁飞区的交点
- 在禁飞区边界外生成绕飞路径
- 绕飞路径需满足最小安全距离要求
- 绕飞路径优化：选择最短绕飞距离方案

#### 4.5.3 路径评估

**FR-NLP-026：航线质量评估指标**

系统应对生成的航线方案计算以下评估指标：

| 评估指标 | 说明 | 计算方式 |
| -------- | ---- | -------- |
| 覆盖率 | 目标区域覆盖率 | 实际覆盖面积 / 目标面积 × 100% |
| 总飞行距离 | 航线总长度（km） | 所有航段距离之和 |
| 预计飞行时间 | 执行航线所需时间 | 距离 / 巡航速度 + 转弯时间 + 悬停时间 |
| 预计电池消耗 | 完成航线所需电量百分比 | 基于能量模型估算 |
| 安全裕度 | 距最近禁飞区/障碍物的最小距离 | min(dist(waypoint, constraint)) |
| 航线效率 | 覆盖面积与飞行距离的比值 | 覆盖面积 / 飞行距离 |
| 约束满足度 | 所有约束的满足程度 | 满足约束数 / 总约束数 × 100% |

**FR-NLP-027：多方案对比输出**

路径规划维应生成不少于3条候选航线方案，各方案在不同优化目标上有所侧重：

- **方案A（效率优先）**：最短飞行时间/距离
- **方案B（质量优先）**：最高覆盖率和重叠率
- **方案C（安全优先）**：最大安全裕度，远离禁飞区和障碍物

### 4.6 第五维：资源调配维

#### 4.6.1 需求概述

资源调配维是五维推理链的最终环节，负责将航线方案与具体设备、时间计划、应急预案组合为完整的可执行任务计划。

#### 4.6.2 设备匹配与分配

**FR-NLP-028：单机任务分配**

对于单机即可完成的任务，系统应评估每个可用设备的适配度：

| 评估维度 | 权重（默认） | 说明 |
| -------- | ------------ | ---- |
| 设备能力匹配度 | 30% | 设备载荷、传感器是否满足任务要求 |
| 续航裕度 | 25% | 预计电池消耗后的剩余电量比例 |
| 设备健康度 | 20% | PHM系统提供的DHI评分 |
| 当前可用性 | 15% | 在线状态、当前位置、最近飞行间隔 |
| 历史适配性 | 10% | 该设备执行同类任务的历史表现 |

**FR-NLP-029：多机任务分解与协调**

对于单机无法完成的任务（区域过大、时间过紧），系统应自动进行多机任务分解：

- **区域分解**：将目标区域划分为N个子区域，每个子区域分配一架无人机
- **时间协调**：编排各架次的起飞时间，避免空域冲突
- **交接设计**：相邻子区域的航线重叠一定间距，确保无遗漏覆盖
- **通信约束**：多机任务时考虑图传频段干扰，合理分配通信信道

#### 4.6.3 时间计划编排

**FR-NLP-030：时间计划生成**

系统应为每个任务计划生成详细的时间线：

```
T-30min  设备预检启动（电池充电确认、传感器校准）
T-15min  航线上传至飞控、参数最终确认
T-5min   气象数据最终检查
T-0      起飞
T+2min   到达目标区域，开始巡检航线
T+20min  航线执行完毕（示例）
T+22min  返航并降落
T+25min  数据回传启动
T+30min  任务完成确认
```

#### 4.6.4 应急预案编排

**FR-NLP-031：应急预案自动生成**

系统应根据环境风险和设备状态，自动生成以下场景的应急预案：

| 应急场景 | 触发条件 | 应急动作 |
| -------- | -------- | -------- |
| 低电量返航 | 电量低于30% | 中断当前航线，执行自动返航，记录已完成覆盖率 |
| 链路中断 | 遥控信号丢失超过10秒 | 执行预设返航航线，尝试重连 |
| 气象恶化 | 风速超过安全阈值或降水开始 | 就近寻找安全降落地点，通知操作员 |
| 禁飞区预警 | 接近禁飞区边界 | 自动偏移航线远离禁飞区，发送告警 |
| 设备故障 | PHM系统告警（EMERGENCY级别） | 立即中断任务，执行应急降落预案 |

#### 4.6.5 最终任务计划输出

### 4.7 推理过程可视化功能

#### 4.7.1 需求概述

系统应提供推理过程的可视化展示，使用户能够理解AI如何从自然语言描述逐步推理出任务方案，增强用户信任度并支持调试和优化。

#### 4.7.2 推理链可视化

**FR-NLP-033：推理流水线可视化**

系统应以流水线图形式展示五维推理链的执行状态：

- 每个维度以节点显示，节点状态：待执行（灰色）、执行中（蓝色脉冲动画）、已完成（绿色）、失败（红色）
- 维度间连线显示数据流向，连线上显示中间结果摘要
- 点击任意已完成节点，展开该维度的详细输入/输出/推理依据
- 整体推理进度条显示当前完成百分比

**FR-NLP-034：中间结果展示**

每个推理维度完成后，系统应即时展示该维度的关键输出：

- 意图解析维：结构化意图卡片（任务类型、区域、时间、设备偏好等）
- 环境感知维：环境仪表盘（气象状况、空域约束数量、风险评分）
- 约束推理维：约束满足矩阵（各约束的满足/冲突/放松状态）
- 路径规划维：地图上的航线预览（多方案叠加对比）
- 资源调配维：最终任务计划摘要卡片

**FR-NLP-035：地图集成展示**

系统应在GIS地图上展示以下信息层：

- 目标区域高亮（蓝色半透明多边形）
- 巡检目标点标注（目标物图标）
- 禁飞区/限高区（红色/橙色半透明多边形）
- 生成的航线（彩色折线，多方案不同颜色）
- 航点标注（序号、高度、速度）
- 预计飞行方向箭头
- 传感器覆盖范围示意（相机FOV投影到地面）

**FR-NLP-036：推理过程回放**

系统应支持推理过程的逐步回放功能：

- 记录每个推理维度的输入、输出、耗时和置信度
- 支持"上一步/下一步"逐步查看推理过程
- 支持"跳转到指定维度"快速定位
- 推理日志支持导出为JSON格式

### 4.8 任务计划管理功能

#### 4.8.1 需求概述

系统应提供任务计划的全生命周期管理能力，支持用户对AI生成的任务计划进行查看、修改、确认、下发等操作。

#### 4.8.2 计划管理操作

**FR-NLP-037：计划列表与查询**

- 列表展示字段：计划名称、任务类型、目标区域、设备、状态、创建时间、推理耗时
- 筛选条件：任务类型、状态、设备、创建时间范围
- 排序：创建时间、推理耗时、约束满足度

**FR-NLP-038：计划修改与反馈**

- 用户可在方案预览阶段对AI生成的计划进行修改：
  - 调整航线（在地图上拖拽航点、新增/删除航点）
  - 更换分配设备
  - 修改执行时间
  - 调整飞行参数（高度、速度）
- 修改后系统应重新执行受影响的推理维度进行可行性校验
- 修改历史记录保留，支持回退到AI原始方案

**FR-NLP-039：计划确认与下发**

- 用户确认计划后，系统将状态变更为CONFIRMED
- CONFIRMED状态的计划可一键下发至任务管理模块创建正式飞行任务
- 下发时自动完成以下校验：
  - 目标设备仍在线且可用
  - 气象条件未恶化（距规划时气象数据变化未超过安全阈值）
  - 禁飞区未发生变化
- 校验不通过时提示用户重新规划或等待条件恢复

```
用户自然语言输入
    │
    ▼
[意图解析维]
    ├── 意图分类 → missionType + confidence
    ├── 实体抽取 → entityList[]
    ├── 约束提取 → constraintList[]
    └── 校验补全 → completeIntentJSON
            │
            ▼
[环境感知维]
    ├── 气象数据 → weatherCondition
    ├── 空域数据 → airspaceConstraints[]
    ├── 地形数据 → terrainProfile
    ├── 设备状态 → deviceAvailability[]
    └── 风险评估 → environmentRiskScore
            │
            ▼
[约束推理维]
    ├── 安全约束 → 安全约束集
    ├── 设备约束 → 设备约束集
    ├── 任务约束 → 任务约束集
    ├── 冲突检测 → conflicts[]
    ├── 冲突消解 → conflictResolutions[]
    └── 可行空间 → feasibleSpace
            │
            ▼
[路径规划维]
    ├── 覆盖算法 → 航线生成
    ├── 航线优化 → 优化航线
    ├── 禁飞规避 → 安全航线
    ├── 质量评估 → 评估指标
    └── 多方案生成 → routeAlternatives[]
            │
            ▼
[资源调配维]
    ├── 设备匹配 → deviceAssignment
    ├── 时间计划 → timePlan
    ├── 应急预案 → emergencyPlans[]
    └── 最终输出 → completeMissionPlan
```
