## 1. 引言

### 1.1 目的

本文档是"基于规则引擎与大模型融合的无人机预测性健康管理系统（PHM）"的软件需求规格说明书（SRS），旨在为系统的设计、开发、测试与验收提供完整、明确、可验证的需求基线。

本文档面向以下读者：

- 系统架构师：理解整体架构约束与技术选型边界
- 后端开发工程师：实现规则引擎、消息队列消费、大模型集成等核心逻辑
- 前端开发工程师：实现规则配置界面、PHM诊断报告展示、DHI健康趋势图表
- 测试工程师：依据本文档制定测试用例和验收标准
- 产品经理与研究人员：作为课题研究与专利申请的需求依据

### 1.2 范围

本系统聚焦以下四个核心能力域：

1. **PHM规则管理**：支持用户通过可视化界面配置、编辑、启用/禁用设备异常检测规则，覆盖阈值类、趋势类、组合类、时序类四种检测模式，针对无人机电池、电机、云台、链路、GPS五大核心组件。

2. **规则引擎实时初筛**：在遥测数据写入流中实时评估所有启用规则，对命中规则的遥测帧生成结构化异常事件，通过消息队列异步下发，不阻塞主数据写入链路。

3. **大模型深度诊断**：消费异常事件队列，融合设备档案、历史工单、同批次DHI对比数据，调用DeepSeek大语言模型完成根因推断，输出包含置信度、紧急程度、维护建议的结构化诊断报告。

4. **设备健康指数（DHI）管理**：基于多维特征计算每台无人机的健康评分，并提供同批次横向对比分析，为AI调度模块的设备权重分配提供数据支撑。

本文档**不覆盖**以下内容：工单管理模块内部逻辑、设备管理基础台账、飞行任务调度系统、视频流与图像AI检测模块。

## 2. 总体描述

### 2.1 产品愿景

无人机在大规模商业巡检与作战集群场景中，设备健康状态直接决定任务成败与人员安全。现有无人机管理系统对设备健康的处置方式存在两个根本性缺陷：

- **被动响应**：设备已发生故障或任务失败后才被发现，缺乏预警能力；
- **单一阈值**：仅依赖固定阈值告警，无法识别渐变型故障（如电池老化、电机磨损）和多因素组合异常。

PHM系统的愿景是构建**主动感知、精准诊断、自动处置**的设备健康管理体系，使无人机集群的计划外停机率降低60%以上，设备维护成本降低40%，同时为AI调度模块提供可信的设备可用性评分，提升任务分配的合理性。

### 2.2 系统架构概述

PHM系统采用"**前端规则管理 → 规则引擎实时初筛 → 消息队列异步解耦 → 大模型深度诊断 → 工单自动创建**"五段式处理流水线：

```
┌───────────────────────────────────────────────────────────┐
│                     前端规则配置界面                        │
│  用户配置规则条件 / 启用·禁用 / 规则测试                    │
└─────────────────────────┬─────────────────────────────────┘
                          │ 规则写入数据库
                          ▼
┌───────────────────────────────────────────────────────────┐
│              规则引擎（遥测数据流实时初筛）                  │
│  数据预处理 → 规则加载(缓存) → 规则评估 → 冷却时间检查      │
└─────────────────────────┬─────────────────────────────────┘
                          │ 仅命中规则的异常帧
                          ▼
┌───────────────────────────────────────────────────────────┐
│                消息队列（phm.anomaly.events）               │
│             异步解耦，削峰填谷，保障写入吞吐量               │
└─────────────────────────┬─────────────────────────────────┘
                          │ 消费者拉取
                          ▼
┌───────────────────────────────────────────────────────────┐
│              大模型诊断服务（DeepSeek）                      │
│  去重检查 → 档案增强 → 提示词构建 → LLM调用 → 报告持久化    │
└─────────────────────────┬─────────────────────────────────┘
                          │ autoCreateWorkorder=true 且 confidence≥60
                          ▼
┌───────────────────────────────────────────────────────────┐
│                 工单管理模块（已有功能）                      │
│             自动创建运维工单 / WebSocket推送前端              │
└───────────────────────────────────────────────────────────┘
```

**核心设计原则**：

- 规则引擎高频轻量，单条遥测帧评估耗时 < 50ms，不阻塞数据写入主链路；
- 大模型诊断低频深度，通过消息队列异步解耦，4小时内同设备同规则只诊断一次；
- 工单由系统在诊断完成后自动创建，人工只需处理工单，无需感知诊断细节。

### 2.3 运行环境

| 层次 | 技术选型 |
| ---- | -------- |
| 后端框架 | Spring Boot 3.x / Java 17 |
| 消息队列 | Apache Kafka 或 RocketMQ（Topic: phm.anomaly.events） |
| 缓存 | Redis 7.x（规则缓存、滑动窗口、冷却时间、去重锁） |

| 前端框架 | Vue 3 / React 18（规则配置页面、PHM仪表板） |
| 实时通信 | WebSocket（PHM诊断完成后推送前端） |

## 4. 功能性需求

### 4.1 PHM规则管理功能

#### 4.1.1 需求概述

系统应提供可视化的规则配置界面，支持运维管理员（R-01）对无人机异常检测规则进行全生命周期管理，无需修改代码或重启服务。

#### 4.1.2 规则数据模型

规则实体应包含以下核心字段：

| 字段名 | 类型 | 说明 | 约束 |
| ------ | ---- | ---- | ---- |
| ruleId | UUID | 规则唯一标识，系统自动生成 | 主键，不可修改 |
| ruleName | String | 规则名称，如"电池低电量告警" | 必填，长度2~50字符 |
| ruleCode | String | 规则编码，如"BATTERY_LOW" | 必填，全局唯一，大写字母+下划线 |
| description | String | 规则描述，在前端列表展示 | 选填，长度≤200字符 |
| category | Enum | 规则分类：THRESHOLD/TREND/COMBINATION/SEQUENCE | 必填 |
| targetScope.deviceTypes | List\<String\> | 适用设备型号列表，空=适用全部 | 选填 |
| targetScope.deviceIds | List\<String\> | 指定设备ID列表，空=适用全部 | 选填 |
| targetScope.component | Enum | 监测组件：BATTERY/MOTOR/GIMBAL/LINK/GPS | 必填 |
| condition.expression | String | 条件表达式，如"battery.level < 20" | 必填 |
| condition.metrics | List\<String\> | 涉及指标列表 | 系统自动解析 |
| condition.timeWindow | Integer | 时间窗口（秒），趋势类必填 | 趋势类必填，≥10 |
| condition.threshold | Double | 阈值数值 | 数值类规则必填 |
| condition.operator | Enum | 比较运算符：LT/GT/LTE/GTE/EQ/NEQ | 阈值类必填 |
| output.severity | Enum | 严重程度：INFO/WARNING/CRITICAL/EMERGENCY | 必填 |
| output.anomalyType | String | 异常类型描述，供大模型识别上下文 | 必填 |
| output.needLLMDiagnosis | Boolean | 是否触发大模型诊断 | 必填，默认false |
| output.autoCreateWorkorder | Boolean | 诊断完成后是否自动创建工单 | 必填，依赖needLLMDiagnosis=true |
| output.workorderPriority | Enum | 工单优先级：LOW/MEDIUM/HIGH/URGENT | autoCreateWorkorder=true时必填 |
| output.cooldownSeconds | Integer | 触发冷却时间（秒），防重复触发 | 必填，范围60~86400 |
| priority | Integer | 规则优先级，1~100 | 必填，默认50 |
| enabled | Boolean | 启用状态 | 必填，默认true |
| version | Integer | 版本号，每次编辑自增 | 系统维护 |
| createdBy | String | 创建人 | 系统自动填入 |
| createdTime | DateTime | 创建时间 | 系统自动填入 |
| updatedTime | DateTime | 最后修改时间 | 系统自动更新 |

#### 4.1.3 规则分类与触发逻辑

**FR-PHM-001：阈值类规则（THRESHOLD）**

系统应支持单指标与固定阈值的比较检测，这是最基础也是使用最频繁的规则类型。

触发判断逻辑：
```
取遥测帧中指定字段值
按 condition.operator 与 condition.threshold 比较
满足条件 → 生成 AnomalyEvent
```

系统应内置以下示例规则（可由管理员修改）：

| 规则编码 | 触发条件 | 严重程度 | 说明 |
| -------- | -------- | -------- | ---- |
| BATTERY_LOW | battery.level < 20 | CRITICAL | 低电量告警 |
| LINK_WEAK | link.rssi < -85 | WARNING | 链路信号弱告警 |
| ALTITUDE_EXCEED | telemetry.altitude > 500 | EMERGENCY | 飞行高度超限 |
| GPS_WEAK | gps.satelliteCount < 6 | WARNING | GPS信号不足 |

**FR-PHM-002：趋势类规则（TREND）**

系统应支持基于滑动窗口的渐变故障检测，适用于电池老化、电机磨损等慢变型故障。

触发判断逻辑：
```
维护设备滑动窗口缓冲（Redis List，长度 = timeWindow / 采样间隔）
窗口满时计算变化率 = (最新值 - 窗口首值) / |窗口首值| × 100%
超过 condition.threshold 定义的变化率阈值 → 触发
```

系统应内置以下示例规则：

| 规则编码 | 触发条件 | 说明 |
| -------- | -------- | ---- |
| BATTERY_DISCHARGE_FAST | battery.level 在10分钟内下降超25% | 电池异常放电 |
| MOTOR_OVERLOAD | motor.current 在5分钟内上升超30% | 电机负载异常 |

**FR-PHM-003：组合类规则（COMBINATION）**

系统应支持多个子条件的逻辑组合检测（AND/OR），以减少单一阈值规则的误报率。

触发判断逻辑：
```
各子条件独立评估
按 AND/OR 逻辑合并结果
最终结果为 true → 触发
```

系统应内置以下示例规则：

| 规则编码 | 触发条件 | 说明 |
| -------- | -------- | ---- |
| MOTOR_VIBRATION_ABNORMAL | motor.vibration > 0.8g AND motor.current > 12A AND speed < 5m/s | 排除正常飞行颠簸，精准识别机械振动 |

**FR-PHM-004：时序类规则（SEQUENCE）**

系统应支持事件时序关系的超时检测，适用于链路中断恢复超时、起飞指令超时等场景。

触发判断逻辑：
```
Redis Hash 记录前置事件A的触发时间
后续数据帧持续检查事件B是否在 condition.timeWindow 内发生
超时未发生 → 触发
```

系统应内置以下示例规则：

| 规则编码 | 触发条件 | 说明 |
| -------- | -------- | ---- |
| LINK_LOST_TIMEOUT | 链路中断后60秒内未恢复 | 触发失联处置工单 |

#### 4.1.4 规则管理界面需求

**FR-PHM-005：规则列表展示**

系统应提供规则列表页面，包含以下元素：

- 规则名称、规则编码、分类、监测组件、严重程度、启用状态、触发次数（近7天）
- 筛选条件：规则分类（多选）、监测组件（多选）、严重程度（多选）、启用状态
- 支持按"触发次数"降序排列，快速定位高频规则
- 启用/禁用开关（行内操作，无需弹窗确认，实时生效）
- 编辑、删除、测试操作按钮

**FR-PHM-006：规则新建/编辑**

系统应提供四步骤的规则配置向导（弹窗形式）：

步骤一：基础信息

- 规则名称（文本输入，必填）
- 规则分类（下拉选择，选后动态渲染步骤二的条件配置区）
- 适用设备型号（多选下拉，空=全部）
- 适用设备范围（可选指定设备ID，支持多选搜索）
- 监测组件（下拉，枚举值：电池/电机/云台/链路/GPS）
- 规则描述（多行文本，选填）

步骤二：触发条件（根据分类动态渲染）

阈值类：
```
[指标下拉] [运算符下拉 (</>/<=/>=) ] [数值输入] [单位显示]
示例：[ battery.level ▼ ] [ < ▼ ] [ 20 ] [ % ]
```

趋势类：
```
[指标下拉] 在 [时间窗口输入] 秒内 [方向下拉(上升/下降/任意)] 超过 [变化率输入] %
```

组合类：
```
条件列表（可增删行）：
  [指标▼] [运算符▼] [数值] [单位] [逻辑符(AND/OR)]
  [指标▼] [运算符▼] [数值] [单位]
  [+ 添加子条件]
```

时序类：
```
当 [前置事件描述] 发生后，[超时时长] 秒内未发生 [后置事件描述]
```

步骤三：告警输出配置

- 严重程度（下拉：INFO/WARNING/CRITICAL/EMERGENCY）
- 异常类型描述（文本输入，供大模型识别上下文，必填）
- 是否触发大模型诊断（开关，默认关闭）
- 诊断完成后自动创建工单（开关，依赖上一项开启时显示）
- 工单优先级（下拉，上一开关开启时显示）
- 冷却时间（滑块 + 数值输入，单位：秒，范围：60~86400）

步骤四：预览确认

- 以自然语言描述规则内容，供用户确认，例如：
  > "当设备电池电量低于20%且正在飞行时，触发CRITICAL级别告警，30分钟内不重复触发，自动创建HIGH优先级工单"
- 按钮：[保存并启用] [保存为草稿] [取消]

**FR-PHM-007：规则测试功能**

系统应提供规则历史数据回放测试能力：

- 用户在规则编辑页面点击"测试规则"入口
- 弹窗：选择目标设备 + 历史时间段（起止时间）
- 系统使用该时间段遥测数据回放，检验规则是否触发
- 返回结果展示：触发时间点列表 + 触发频次统计 + 触发示例数据
- 测试过程不产生实际告警，不创建工单

**FR-PHM-008：规则导入/导出**

- 支持将规则配置导出为JSON格式文件
- 支持从JSON文件导入规则配置（重名规则弹窗询问覆盖或跳过）
- 导入前进行格式和字段合法性校验，校验失败显示错误行号和原因

### 4.2 规则引擎实时初筛功能

#### 4.2.1 需求概述

规则引擎是PHM系统的实时数据处理核心，在每条遥测数据写入时同步执行规则评估，但不阻塞数据写入主链路。

#### 4.2.2 数据预处理需求

**FR-PHM-009：遥测数据字段规范化**

系统应在规则评估前对遥测数据执行以下预处理：

- 单位统一：电压统一为V，电流统一为A，温度统一为℃，速度统一为m/s
- 合法性校验：超出物理范围的值直接丢弃（如电量 > 100%、高度 < -100m）
- 字段缺失处理：必需字段缺失时跳过相关规则，不触发告警

#### 4.2.3 规则加载与缓存需求

**FR-PHM-010：规则缓存机制**

- 规则引擎从本地缓存加载当前设备适用的启用规则集
- 缓存Key策略：按设备型号+组件分类建立规则索引缓存
- 缓存TTL：5分钟自动过期刷新
- 规则变更时（新增/修改/启用/禁用）：主动清除相关缓存，确保实时生效，延迟 < 5秒

#### 4.2.4 规则评估需求

**FR-PHM-011：四类规则评估逻辑**

规则引擎应按如下逻辑分类评估：

| 规则类型 | 评估逻辑 | 数据来源 |
| -------- | -------- | -------- |
| 阈值类 | 直接计算条件表达式 | 当前遥测帧 |
| 趋势类 | 从Redis读取滑动窗口数据后计算变化率 | Redis滑动窗口 + 当前帧 |
| 组合类 | 子条件分别评估，合并AND/OR逻辑 | 当前遥测帧 |
| 时序类 | 查询Redis状态机，判断超时 | Redis状态机 |

**FR-PHM-012：冷却时间控制**

- Redis Key格式：`phm:cool:{ruleId}:{deviceId}`
- Key TTL = cooldownSeconds（规则配置的冷却时间）
- 命中冷却期的触发事件直接丢弃，不进入消息队列
- 同一规则同一设备在冷却期内最多触发一次告警

**FR-PHM-013：AnomalyEvent生成**

命中规则且通过冷却检查后，系统应生成如下结构的异常事件对象并写入消息队列：

| 字段 | 类型 | 说明 |
| ---- | ---- | ---- |
| eventId | UUID | 异常事件唯一标识 |
| deviceId | String | 设备ID |
| deviceName | String | 设备名称 |
| ruleId | String | 命中规则ID |
| ruleCode | String | 规则编码 |
| severity | Enum | 严重程度 |
| anomalyType | String | 异常类型描述 |
| triggerTime | DateTime | 触发时间 |
| telemetrySnapshot | Object | 触发时遥测快照（见4.6节） |
| context | Object | 上下文信息（见4.6节） |
| needLLMDiagnosis | Boolean | 是否需要大模型诊断 |
| autoCreateWorkorder | Boolean | 是否自动创建工单 |
| workorderPriority | Enum | 工单优先级 |

### 4.3 大模型深度诊断功能

#### 4.3.1 需求概述

大模型诊断服务作为消息队列的消费者异步运行，对命中规则的异常事件进行深度根因分析，输出结构化诊断报告，并依据配置决定是否自动创建工单。

#### 4.3.2 消费处理流程

**FR-PHM-014：去重检查**

- Redis Key格式：`phm:diag:dedup:{deviceId}:{ruleCode}`，TTL = 4小时
- 4小时内同设备同规则只触发一次大模型诊断
- 命中去重锁的事件跳过大模型调用，在日志中记录去重原因

**FR-PHM-015：设备档案增强**

在调用大模型前，系统应从数据库补充以下增强信息：

- 设备完整信息（型号、累计飞行时长、上次维护日期）
- 近30天工单记录（数量和摘要）
- 近7天同类型告警历史（次数）
- 同批次设备的DHI统计（平均值、本机排名）

### 4.4 设备健康指数（DHI）管理功能

#### 4.4.1 需求概述

DHI（Device Health Index）是对单台无人机综合健康状况的量化评分（0~100分），为设备状态可视化和AI智能调度提供统一的健康基准。

#### 4.4.2 DHI计算需求

**FR-PHM-020：DHI多维度计算模型**

系统应基于以下维度计算DHI，各维度权重可由系统管理员配置（默认值如下）：

| 维度 | 权重（默认） | 计算说明 |
| ---- | ------------ | -------- |
| 电池健康度 | 30% | 基于充放电循环次数、电池内阻变化率、放电曲线偏差 |
| 电机健康度 | 25% | 基于振动频谱分析、电流基线偏差、转速均衡性 |
| 链路稳定性 | 20% | 基于RSSI趋势、丢包率历史、连接中断频次 |
| 飞行可靠性 | 15% | 基于近30天任务成功率、异常中断次数 |
| 维护及时性 | 10% | 基于距上次维护时间、当前工单积压数量 |

**FR-PHM-021：DHI更新触发机制**

DHI应在以下场景触发重新计算：

- 每次飞行任务完成后（飞行数据对各维度评分有更新）
- 大模型诊断报告确认后（维护状态变更）
- 工单完成后（维护及时性维度更新）
- 定时任务：每日凌晨2:00全量更新所有设备DHI

**FR-PHM-022：DHI历史趋势记录**

系统应持久化每次DHI计算结果，记录包含：

- 设备ID、计算时间、DHI总分、各维度分项得分
- 支持按设备ID查询任意时间段的DHI历史曲线
- 历史数据保留周期：至少12个月

#### 4.4.3 DHI仪表板展示需求

**FR-PHM-023：设备健康总览**

系统应提供PHM仪表板页面，展示以下内容：

- 机队整体健康状态统计（健康/警告/危险设备数量，可点击钻取）
- 健康等级划分标准：DHI ≥ 80为健康（绿色），60~79为警告（黄色），< 60为危险（红色）
- 近期PHM诊断报告数量统计（按严重程度分色）
- 待处理工单数量（来源PHM_AUTO的工单）

**FR-PHM-024：单设备健康详情**

用户点击设备进入健康详情页，展示以下内容：

- DHI历史趋势折线图（近90天，每日一个数据点）
- 各维度分项得分雷达图（当前最新值）
- 同批次设备DHI对比箱线图（本机当前DHI在同批次分布中的位置）
- 近期PHM诊断报告列表（时间倒序，含状态和置信度）
- 在飞诊断报告时对应的历史工单列表

## 8. 约束与假设

### 8.1 技术约束

| 约束编号 | 约束内容 |
| -------- | -------- |
| CON-001 | 规则引擎必须与遥测数据写入链路同进程部署，不允许跨网络调用，确保评估延迟可控 |
| CON-002 | 大模型诊断服务必须与规则引擎异步解耦，不允许同步等待LLM响应 |
| CON-003 | 系统依赖DeepSeek Chat API的外部可用性，LLM接口SLA应≥99.5%，否则需配置降级方案 |
| CON-004 | Redis作为规则缓存、滑动窗口、冷却时间的统一存储介质，不可替换为本地内存缓存（无法支持多实例部署） |
| CON-005 | DHI计算不得在遥测数据主写入路径中执行，应通过独立定时任务或事件触发异步计算 |

### 8.2 业务约束

| 约束编号 | 约束内容 |
| -------- | -------- |
| CON-006 | PHM规则不可物理删除，仅可禁用，以保证历史告警事件的溯源完整性 |
| CON-007 | 诊断报告一经生成，内容不可修改，仅允许运维管理员通过"确认/驳回"操作变更处理状态 |
| CON-008 | 自动创建的工单必须标注来源字段source="PHM_AUTO"，与人工创建工单区分 |
| CON-009 | 置信度 < 60的诊断结果不得自动创建工单，只能由人工审核后手动触发 |

### 8.3 假设条件

| 假设编号 | 假设内容 |
| -------- | -------- |
| ASS-001 | 无人机遥测数据上报频率不低于1Hz，能够满足滑动窗口趋势计算的数据密度要求 |
| ASS-002 | 工单管理模块提供稳定的创建工单API，接口响应时间 < 2秒 |
| ASS-003 | 同批次设备的DHI对比计算以"设备出厂批次号"作为分组依据，该字段已在设备管理模块中维护 |
| ASS-004 | DeepSeek Chat API在中国大陆网络环境下可稳定访问，平均响应时间 < 15秒 |
| ASS-005 | 系统运行在Linux服务器环境（CentOS 7+或Ubuntu 20.04+），Redis和消息队列已预先部署 |
