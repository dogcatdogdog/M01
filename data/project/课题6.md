## 1. 引言

### 1.1 目的

本文档是"异构无人机协议标准化SDK的插件化适配器框架设计"的软件需求规格说明书（SRS），旨在为系统的设计、开发、测试与验收提供完整、明确、可验证的需求基线。

本文档面向以下读者：

- 系统架构师：理解插件化框架的架构约束与扩展机制设计边界
- SDK开发工程师：实现标准化接口定义、适配器框架核心、生命周期管理
- 协议开发工程师：基于SDK开发具体设备协议适配器
- 第三方开发者：作为SDK二次开发和适配器开发的技术参考
- 产品经理与研究人员：作为课题研究与专利申请的需求依据

### 1.2 范围

本系统聚焦以下五个核心能力域：

1. **标准化设备抽象层**：定义统一的无人机设备抽象模型（UDI），涵盖设备连接、遥测数据、航线控制、载荷控制、参数管理五大接口域

2. **插件化适配器框架**：设计"适配器接口规范→注册中心→生命周期管理→热插拔"完整插件化架构，新设备接入"零侵入"

3. **协议转换引擎**：异构协议消息到标准化接口的双向转换，支持MAVLink v2、DJI Cloud API/PSDK等，提供通用协议描述语言（PDL）

4. **设备能力发现与协商**：设备能力自描述与自动发现，框架根据能力动态调整上层功能可用性

5. **适配器开发工具链**：脚手架、协议调试、一致性测试、性能基准等全流程工具

本文档**不覆盖**：具体设备适配器完整实现、飞控固件开发、硬件驱动开发、上层业务逻辑。

## 2. 总体描述

### 2.1 产品愿景

异构无人机设备接入存在三个根本性问题：

- **协议碎片化**：各厂商使用不同协议，新设备接入需2~4周，维护成本指数上升
- **接口不统一**：上层代码直接调用特定协议，设备更换需大面积修改
- **开发门槛高**：新适配器开发者需深入理解平台架构和协议细节

SDK愿景：构建**"一套接口、插件接入、即插即用"**的标准化适配框架，新设备接入从2~4周缩短至2~3天，上层代码与底层协议完全解耦。

**课题创新性**：

1. **UDI统一接口抽象**：覆盖连接、遥测、航线、载荷、参数五大维度的协议无关"最大公约数"接口模型
2. **能力协商与动态适配**：借鉴Capability Exchange思想，设备能力自描述与自动协商
3. **协议描述语言（PDL）**：声明式配置定义协议映射，简单协议可"零代码适配"
4. **开发工具链一体化**：从脚手架到性能基准的全流程工具

### 2.2 系统架构概述

SDK采用**"核心层→接口层→适配器层→工具层"**四层架构：

```
┌─────────────────────────────────────────────────────────────┐
│                       上层业务应用                            │
│   任务管理 / 实时监控 / 飞行控制 / PHM / AI决策引擎            │
└─────────────────────────┬───────────────────────────────────┘
                          │ 统一无人机接口（UDI）
                          ▼
┌─────────────────────────────────────────────────────────────┐
│              接口层：UDI接口集                                │
│  DeviceConnectionAPI / TelemetryAPI / WaylineAPI            │
│  PayloadAPI / ParamAPI / EventAPI / CapabilityAPI           │
└─────────────────────────┬───────────────────────────────────┘
                          │ 接口调用
                          ▼
┌─────────────────────────────────────────────────────────────┐
│              核心层：适配器框架内核                            │
│  注册中心 / 生命周期管理器 / 能力协商引擎 / 协议转换引擎       │
│  事件总线 / 线程模型管理 / 配置管理器 / 监控与度量             │
└─────────────────────────┬───────────────────────────────────┘
                          │ 标准化适配器接口（SPI）
                          ▼
┌─────────────────────────────────────────────────────────────┐
│              适配器层：具体协议适配器                          │
│  MAVLink Adapter / DJI Cloud Adapter / Autel Adapter        │
│  自定义协议 Adapter    [JAR/Bundle动态加载]                   │
└─────────────────────────┬───────────────────────────────────┘
                          ▼
┌─────────────────────────────────────────────────────────────┐
│              工具层：适配器开发工具链                          │
│  脚手架生成器 / 协议调试器 / 一致性测试 / 性能基准工具         │
└─────────────────────────────────────────────────────────────┘
```

**核心设计原则**：
- **接口隔离**：上层仅依赖UDI接口，不引用具体适配器
- **开闭原则**：新适配器不修改框架核心代码，SPI动态加载
- **能力驱动**：调用前查询设备能力，不支持的功能直接拒绝
- **故障隔离**：单适配器异常不影响框架核心和其他适配器

## 4. 功能性需求

### 4.1 统一无人机接口（UDI）定义

#### 4.1.1 需求概述

UDI是SDK核心接口集，定义上层应用与无人机交互的标准化方法。所有适配器必须实现UDI，所有上层应用必须通过UDI访问设备。

#### 4.1.2 接口域划分

**FR-SDK-001：设备连接接口（DeviceConnectionAPI）**

核心方法：connect（建立连接）、disconnect（断开）、getConnectionState（状态查询）、addConnectionStateListener（监听）、sendHeartbeat（心跳）。

连接状态：DISCONNECTED→CONNECTING→AUTHENTICATING→CONNECTED，异常时RECONNECTING/ERROR。

连接配置（ConnectionConfig）：protocolType（协议标识）、transportType（SERIAL/UDP/TCP/USB/4G_LTE/WIFI）、address（连接地址）、credentials（认证信息）、reconnectPolicy（重连策略）、timeout（超时）、customParams（自定义参数）。

**FR-SDK-002：遥测数据接口（TelemetryAPI）**

核心方法：subscribe（订阅数据流）、unsubscribe（取消）、getLatestSnapshot（最新快照）、queryHistory（历史查询）、getSupportedFields（支持字段列表）。

标准化遥测数据模型（TelemetrySnapshot）字段组：

| 字段组 | 核心字段 | 类型 | 单位 |
| ------ | -------- | ---- | ---- |
| 姿态 | roll/pitch/yaw | Float | ° |
| 位置 | latitude/longitude/altitude/relativeAltitude | Double/Float | °/m |
| 速度 | vx/vy/vz/groundSpeed | Float | m/s |
| 电池 | level/voltage/current/temperature/cycleCount | Integer/Float | %/V/A/℃ |
| GPS | fixType/satellites/hdop | Enum/Integer/Float | — |
| 链路 | rssi/snr/packetLoss | Float | dBm/dB/% |
| 电机 | motors[].rpm/current/temperature | Integer/Float | rpm/A/℃ |
| 飞控 | flightMode/armed/isFlying | Enum/Boolean | — |
| 环境 | windSpeed/temperature | Float | m/s/℃ |
| 时间戳 | timestamp | Long | ms |

**FR-SDK-003：航线控制接口（WaylineAPI）**

核心方法：uploadWayline（上传）、startWayline/pauseWayline/resumeWayline/abortWayline（执行控制）、getProgress（进度）、getWaylineConstraints（约束查询）。

航线数据模型（WaylineData）：waylineId、name、type（WAYPOINT/AREA_SURVEY/CORRIDOR）、waypoints（航点列表）、globalSpeed、globalAltitude、finishAction（HOVER/LAND/RETURN_HOME）、lostLinkAction。

航点模型（Waypoint）：index、latitude、longitude、altitude、speed、heading、gimbalPitch、actions（TAKE_PHOTO/START_RECORD/STOP_RECORD/ROTATE_GIMBAL/ADJUST_SPEED/HOVER）、stayTime、cornerRadius。

**FR-SDK-004：载荷控制接口（PayloadAPI）**

核心方法：takePhoto（拍照）、startRecording/stopRecording（录像）、setGimbalAngle（云台）、controlPayload（挂载控制）、getPayloads（载荷列表）、getCameraCapabilities（相机能力）。

**FR-SDK-005：参数管理接口（ParamAPI）**

核心方法：getParam/getParams（获取参数）、setParam（设置）、getParamDescriptors（参数描述）、addParamChangeListener（变更监听）。

**FR-SDK-006：事件接口（EventAPI）**

核心方法：subscribeDeviceEvents（设备事件订阅）、subscribeGlobalEvents（全局事件）、unsubscribe（取消）。

标准化设备事件：CONNECTION_STATE_CHANGED、BATTERY_LOW、GPS_SIGNAL_CHANGED、LINK_QUALITY_CHANGED、MODE_CHANGED、GEOFENCE_BREACH、MOTOR_ANOMALY、WAYPOINT_REACHED、MISSION_COMPLETED、FAILSAFE_TRIGGERED、OBSTACLE_DETECTED。

**FR-SDK-007：设备能力接口（CapabilityAPI）**

核心方法：getCapabilities（完整能力模型）、isFeatureSupported（功能检查）、getFeatureRange（参数范围）、addCapabilityChangeListener（变更监听）。

能力模型（DeviceCapability）能力组：

| 能力组 | 核心字段 |
| ------ | -------- |
| 飞行能力 | maxAltitude/maxSpeed/maxEndurance/maxWaypoints/hoverCapability/precisionLanding |
| 传感器能力 | cameras/hasThermalCamera/hasRTK/hasLiDAR/hasObstacleAvoidance |
| 通信能力 | protocols/maxBandwidth/linkTypes |
| 载荷能力 | maxPayloadWeight/payloadSlots |
| 安全能力 | failsafeActions/geofenceSupport/maxWindResistance |

#### 4.1.3 接口版本管理

**FR-SDK-008：UDI接口版本控制**

采用语义版本号（SemVer）：Major不兼容变更、Minor向后兼容新增、Patch修复。同时维护最多2个Major版本，旧版本标记Deprecated。

### 4.2 插件化适配器框架

#### 4.2.1 需求概述

插件化框架实现适配器的动态发现、加载、管理和卸载，新设备接入无需修改框架核心代码。

#### 4.2.2 适配器SPI规范

**FR-SDK-009：适配器接口规范**

每个适配器实现DroneAdapter接口，包含：元信息（adapterId/version/displayName/protocolType/supportedDeviceModels）、七大接口域实现（ConnectionAPI/TelemetryAPI/WaylineAPI/PayloadAPI/ParamAPI/EventAPI/CapabilityAPI）、生命周期方法（initialize/start/stop/destroy）、健康检查（getHealthStatus）。

**FR-SDK-010：适配器注册方式**

三种注册方式：Java SPI自动发现（META-INF/services声明）、编程式注册（AdapterRegistry.getInstance().register()）、配置文件声明（adapters.yaml）。

#### 4.2.3 适配器生命周期管理

**FR-SDK-011：适配器生命周期状态机**

6个状态：UNREGISTERED→REGISTERED→INITIALIZED→RUNNING→STOPPED→DESTROYED，任意状态可转ERROR。

| 事件 | 触发条件 | 操作 |
| ---- | -------- | ---- |
| register | SPI发现/编程注册/配置加载 | 加载类，实例化 |
| initialize | 注册后触发 | 注入Context，加载配置 |
| start | 初始化后触发 | 启动通信和线程池 |
| stop | 手动/异常 | 停止通信，释放连接 |
| destroy | 手动/系统关闭 | 释放所有资源 |
| unregister | 手动注销 | 从注册中心移除 |

**FR-SDK-012：适配器热插拔**

- **热加载**：文件监听器监控JAR目录，新增自动register→initialize→start
- **热卸载**：等待活跃连接断开（超时30秒强制）
- **热更新**：先加载新版本→停旧版本→启新版本，无缝切换
- **故障隔离**：单适配器异常不传播到框架核心

#### 4.2.4 适配器注册中心

**FR-SDK-013：注册中心功能**

功能：适配器注册/注销（校验接口合规性）、查询（按adapterId/protocolType/deviceModel）、能力路由（设备型号匹配）、优先级管理、版本管理、全局单例多线程安全。

**FR-SDK-014：设备-适配器路由规则**

路由流程：查询设备协议类型→查找支持该协议的适配器→按优先级排序→选最高优先级RUNNING适配器→无匹配抛UnsupportedDeviceException。

### 4.3 协议转换引擎

#### 4.3.1 需求概述

协议转换引擎负责底层异构协议消息与UDI标准化数据的双向转换，是每个适配器的核心工作。

#### 4.3.2 转换架构

**FR-SDK-015：双向转换模型**

出站转换（Outbound）：UDI指令→协议消息→发送到设备。入站转换（Inbound）：设备接收→协议消息→UDI标准化数据→通过EventAPI传递。

**FR-SDK-016：转换器开发接口**

SDK提供三类转换器基类：

- **TelemetryConverter**：协议消息→UDI遥测快照，子类实现mapFields方法
- **WaylineConverter**：UDI航线→协议航线格式，子类实现encodeWayline方法
- **CommandConverter**：UDI指令→协议指令，包含convertTakeoffCommand/convertLandCommand/convertReturnHomeCommand/convertCustomCommand

#### 4.3.3 协议描述语言（PDL）

**FR-SDK-017：协议描述语言设计**

对简单协议，通过YAML配置文件定义协议映射，无需编写Java代码。PDL定义包含：

- **传输层配置**：类型（UDP/TCP/Serial）、端口、帧格式（起始字节、长度字段）
- **消息类型定义**：消息ID、字段列表（名称、类型、偏移、缩放因子、枚举映射）
- **UDI映射**：协议字段→UDI字段的映射关系（如"position.latitude"←"GPS_POSITION.latitude"）
- **指令映射**：UDI指令→协议消息的映射（含payload格式定义）

**FR-SDK-018：PDL编译器**

PDL编译器将YAML编译为：Java适配器代码骨架（辅助开发）、运行时协议描述对象、协议文档（Markdown）。支持高级特性：消息聚合（多消息→单快照）、字段转换函数（scale/offset/enum/位操作）、条件映射、默认值。

### 4.4 设备能力发现与协商

#### 4.4.1 需求概述

设备能力发现与协商使框架自动感知设备支持的功能范围，根据能力动态调整上层功能可用性。

#### 4.4.2 能力发现流程

**FR-SDK-019：能力自动发现**

设备连接后自动执行四步发现流程：

1. **基础能力探测**：协议版本查询、设备型号/固件版本、支持的消息类型
2. **功能能力查询**：支持的命令/遥测字段/载荷类型/航线格式和参数范围
3. **约束能力获取**：最大高度/速度/续航、最大航点数、安全保护动作、通信特征
4. **能力模型构建**：合并信息构建DeviceCapability，注册到能力注册表，触发CAPABILITY_DISCOVERED事件

**FR-SDK-020：能力降级与兼容**

| 不支持功能 | 降级策略 |
| ---------- | -------- |
| 悬停 | 航线暂停改为返航替代 |
| 精准降落 | 降落精度标注"普通级" |
| RTK | 定位精度标注"GPS级" |
| 热成像 | 相关载荷功能标记不可用 |
| 避障 | 安全距离约束自动增大 |
| 云台控制 | 航线中云台动作自动跳过 |

上层调用不支持功能时返回UnsupportedCapabilityException并附带降级建议。

### 4.5 适配器开发工具链

#### 4.5.1 需求概述

SDK提供完整开发工具链，覆盖项目创建到发布部署全流程。

#### 4.5.2 脚手架生成器

**FR-SDK-021：适配器项目脚手架**

## 8. 约束与假设

### 8.1 技术约束

| 编号 | 约束内容 |
| ---- | -------- |
| CON-001 | SDK使用Java 17+，不可使用Virtual Threads等新特性 |
| CON-002 | UDI接口签名不得出现特定协议类型或常量 |
| CON-003 | 适配器类加载器与框架核心隔离，防止依赖冲突 |
| CON-004 | 遥测转换在单线程内完成，不可跨线程传递中间状态 |
| CON-005 | SDK核心不引入具体协议编解码库 |
| CON-006 | PDL生成代码必须通过与手写代码相同的一致性测试 |

### 8.2 业务约束

| 编号 | 约束内容 |
| ---- | -------- |
| CON-007 | 认证信息由管理员统一配置，不可硬编码 |
| CON-008 | 有设备连接的适配器不可直接卸载 |
| CON-009 | SDK升级保持至少一个Major版本向后兼容 |
| CON-010 | 一致性测试不通过不可发布生产环境 |

### 8.3 假设条件

| 编号 | 假设内容 |
